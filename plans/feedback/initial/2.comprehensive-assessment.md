## 2. AREAS FOR IMPROVEMENT

### 2.1 Critical (Must Fix Before Building)

| # | Issue | Risk | Recommendation |

| C1 | **No error boundary strategy in shell** | First crash kills the demo | Add React Error Boundaries per panel + per section from Phase 1.0 — not Phase 4 |
| C2 | **Listen Mode overlay scope conflict** | Google feedback says "left panel only" but Doc 5 says "both left and center" | Resolve: Google feedback is correct. Listen overlay covers left panel. Center canvas gets ambient orb glow only (never obscures Reality preview) |
| C3 | **framer-motion mentioned in Google feedback but NOT in approved deps** | Agents will install unapproved package | Decision needed: Either add `framer-motion` to approved list (recommended for mode transitions) OR implement transitions with CSS-only (Tailwind animate + transitions) |

## 4. UX DESIGN ASSESSMENT

### 4.1 Information Architecture

**Strengths:**
- Two-axis mode system creates clear mental model
- Four center tabs map to four user needs: see it, inspect it, document it, understand the AI
- Left panel as "commander" (input), Center as "canvas" (output), Right as "inspector" (detail)

**Issues:**
- **Tab overflow risk**: Four tabs + sub-toggles (HUMAN/AISP in XAI Docs) creates 5 distinct views. Users may not discover XAI Docs or Workflow
- **Recommendation**: Add subtle pulsing dot on XAI DOCS tab when specs auto-update. Add notification badge on WORKFLOW tab when pipeline is active

### 4.2 Interaction Design

**Strengths:**
- Mode switching preserves all state (critical for trust)
- Vibe = one-click macro that changes 15+ nodes (brilliant simplification)
- Click-to-select in Reality tab is intuitive

**Issues:**
- **No undo confirmation for destructive actions** (vibe switch, section delete, reset)
- **No hover preview for vibes** — user must click to see the effect, which triggers a state change
- **Chat input competes with left panel for attention** — in DRAFT mode, the left panel has vibes + sections + chat
- **Recommendation**: Add hover preview for vibes (temporary preview, reverts on mouse-leave). Add toast confirmation for destructive actions. Consider floating chat input at bottom of screen instead of embedded in left panel

### 4.3 Visual Design

**Strengths:**
- "Warm Precision" aesthetic is distinctive and achievable with Tailwind
- Monospace labels create engineering credibility without looking cold
- Orange accent is unique in the AI product space
- Listen Mode dark overlay creates genuine theatrical contrast

**Issues:**
- **Contrast ratios in warm chrome need verification**: `#8a7a6d` (secondary text) on `#faf8f5` (background) = 3.76:1 — **FAILS WCAG AA** for normal text (needs 4.5:1). Fix: darken secondary to `#7a6a5d` (4.53:1) or use only for large text
- **Monospace at 11px is very small**: On low-DPI screens, JetBrains Mono at 11px may be unreadable. Set minimum display size to 12px for all monospace text, with 11px only for status bar
- **Orange on cream has low contrast**: `#e8772e` on `#faf8f5` = 3.21:1 — **FAILS WCAG AA**. Fix: use orange only on white/dark backgrounds, never on cream. Or darken accent to `#d46820` (3.89:1) — still borderline. Best: use orange for large text/icons only, not small text on cream
- **Recommendation**: Run full WCAG contrast audit on the 28-color palette before Phase 1.0. Fix contrast failures in `tailwind.config.ts` tokens

### 4.4 Motion & Animation Design

**Strengths:**
- Red orb 3-layer glow is well-specified
- Breathing animation creates organic, living feel
- Status-based pulse speed (IDLE vs THINKING vs LISTENING) is functional, not decorative

**Issues:**
- **No transition spec for mode switching**: DRAFT → EXPERT should have a subtle panel width + typography density shift (150-200ms ease). Currently no animation defined
- **No transition spec for Listen overlay**: 300ms fade is specified but needs easing curve. Recommend `cubic-bezier(0.4, 0, 0.2, 1)` (Tailwind's default ease-out)
- **Tab switching has no animation**: Add 150ms fade or slide for tab content changes
- **Recommendation**: Define a motion spec: `--transition-fast: 150ms ease`, `--transition-normal: 300ms ease-out`, `--transition-slow: 500ms cubic-bezier(0.4, 0, 0.2, 1)`. Apply consistently

### 4.5 Responsive Design

**Strengths:**
- Desktop-only tool (like Figma, Framer, Webflow) — correct scoping
- Device preview toggle handles the "responsive" need without responsive UI

**Issues:**
- **No minimum viewport width specified**: 3-panel layout at 1024px will be cramped. Define minimum: 1280px. Below that, show "Best experienced on a wider screen" message
- **Panel collapse behavior undefined**: What happens when left panel is collapsed? Does the toggle still show? Does the status bar change?
- **Recommendation**: Define collapse states for each panel. Left panel collapses to 48px icon strip. Right panel collapses to 0px (hidden). Center always visible

## 5. UI/UX TESTING STRATEGY

### 5.1 Recommended Testing Architecture

```
Testing Pyramid for Hey Bradley
================================

                    /\
                   /  \         Playwright E2E + Visual Regression
                  / E2E \       (dual modes, responsive, animations)
                 /________\
                /          \    Playwright Screenshots + Claude Vision AI
               /  Visual    \   (orb glow quality, design compliance)
              /______________\
             /                \  Vitest + React Testing Library
            /   Component      \ (logic, state, data flow, JSON sync)
           /____________________\
          /                      \ axe-core (a11y in both modes)
         /     Accessibility      \ eslint-plugin-tailwindcss (tokens)
        /__________________________\
       /                            \ Vitest unit tests
      /         Unit / Schema        \ (Zustand stores, deepMerge, Zod)
     /________________________________\
```

### 5.2 Tool Recommendations

| Category | Primary Tool | Why |
|----------|-------------|-----|
| **Visual Regression** | **Playwright `toHaveScreenshot()`** | Free, zero vendor lock-in, handles dual visual modes natively, `animations: 'disabled'` freezes orb for deterministic comparisons |
| **Component Testing** | **Vitest + React Testing Library** | Shares Vite config, 50x faster than browser-based tests, perfect for Zustand store + JSON sync logic |
| **E2E Testing** | **Playwright** | Real browser rendering for CSS animations, responsive breakpoints, hover states. Multi-browser (Chrome/Firefox/Safari) from one config |
| **AI Visual Evaluation** | **Custom Claude Vision** | No off-the-shelf tool can evaluate "does the orb have 3 visible glow layers?" or "is this enterprise-dense but readable?" — needs vision model with specific prompts |
| **Accessibility** | **axe-core via `@axe-core/playwright`** | Integrates into existing Playwright tests with one line. Tests both warm chrome and dark Listen mode contrast ratios |
| **Design Token Compliance** | **eslint-plugin-tailwindcss** | Static analysis at save-time. Catches arbitrary values that bypass design tokens |
| **Interaction Testing** | **Playwright** | Only real browser can test CSS transitions, hover: variants, and animation keyframes |

### 5.3 AI-Powered Visual Testing (Unique to This Project)

Standard pixel-diff tools can't evaluate:
- "Is the red orb glow layered correctly?"
- "Does this property inspector feel enterprise-dense but still readable?"
- "Is the warm-to-dark mode transition smooth?"

**Approach**: Playwright takes screenshots → Claude Vision evaluates against criteria:

```typescript
// Example: Evaluate orb quality
test('orb renders with 3 glow layers', async ({ page }) => {
  await page.goto('/builder');
  await page.click('[data-testid="listen-toggle"]');
  await page.waitForTimeout(500); // let animation settle
  const screenshot = await page.locator('.orb-container').screenshot();

  const evaluation = await evaluateWithClaude(screenshot,
    'Verify: 1) bright red inner core, 2) medium diffuse glow ring, ' +
    '3) subtle outer ambient glow. All concentric and smoothly blended.'
  );
  expect(evaluation.pass).toBe(true);
});
```

## 6. RISK MATRIX

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| **Bidirectional JSON editing causes infinite loops** | Medium | High | Use `source` parameter in configStore to break cycles. JSON editor changes tagged as "json-editor", preview changes tagged as "ui" |
| **3-layer orb glow performs poorly on low-end hardware** | Medium | High | CSS `will-change: transform, opacity` on orb layers. Reduce blur radius on lower-spec devices via media query `(prefers-reduced-motion)` |
| **Enterprise density is unreadable** | Medium | Medium | Test at 100% zoom on 1080p display. If 11px mono is too small, floor at 12px. User test with 2-3 people |
| **LLM patches corrupt state (Phase 5)** | High | Critical | Zod validation on every patch BEFORE merge. Reject invalid patches with toast notification. Undo always available |
| **Demo fails on presentation day** | Low | Critical | Build "demo mode" with pre-recorded pipeline states. Workflow tab shows canned data. Listen mode works with pre-recorded transcript |
| **WCAG contrast failures** | High | Medium | Run axe-core in CI from Phase 1.0. Fix contrast issues in design tokens before building components |
| **Vibe switch destroys custom work** | Medium | Medium | Add confirmation. Consider "apply to new sections only" option |
