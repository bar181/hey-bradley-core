# Hey Bradley MVP — Dependency Audit & Recommendations

**Version:** 1.0 | **Date:** March 28, 2026 | **Author:** Pre-Build Assessment
**Goal:** Minimal dependencies. Every package must justify its existence.

---

## Guiding Principles

1. **Fewer dependencies = fewer problems.** Every package is a maintenance liability.
2. **Browser APIs first.** Use native APIs before reaching for a library.
3. **Copy > Install.** shadcn/ui pattern — copy source into project, don't install.
4. **No zip library needed.** Spec download can use individual file downloads or a simple JSON blob.

---

## 1. MANDATORY — Core Platform (Cannot Build Without)

These are non-negotiable. They define the stack.

| Package | Purpose | Size | Notes |
|---------|---------|------|-------|
| `react` + `react-dom` | UI framework | ~140KB | The foundation |
| `vite` + `@vitejs/plugin-react` | Build tool / dev server | Dev only | Fast HMR, zero-config |
| `typescript` | Type safety | Dev only | Strict mode required |
| `tailwindcss` + `postcss` + `autoprefixer` | Styling | Dev only (CSS output) | Utility-first, JSON-mappable |

**Total mandatory runtime deps: 2** (react, react-dom)

---

## 2. RECOMMENDED — High Value, Low Cost

Strong justification. Would be painful to reimplement.

| Package | Purpose | Size | Mandatory? | Alternative |
|---------|---------|------|:---:|-------------|
| `zustand` | State management (configStore, uiStore) | ~2KB | **YES** | None realistic — Redux is heavier, Context is insufficient for this architecture. ADR-005 mandates Zustand. |
| `zod` | Schema validation + type inference | ~13KB | **YES** | None realistic — runtime validation of JSON from DATA tab edits and LLM patches requires a schema library. Zod gives both validation AND TypeScript types from one source. |
| `react-router-dom` | SPA routing (/, /builder, /builder/:id) | ~15KB | **YES** for L2+ | Could use hash-based routing for L1 MVP (single page), but onboarding page in L2 needs real routing. |

---

## 3. OPTIONAL — Evaluate Carefully

Each has a viable "build it yourself" alternative. Decide per-package.

### 3a. UI & Layout

| Package | Purpose | Size | Recommended? | Alternative |
|---------|---------|------|:---:|-------------|
| `react-resizable-panels` | Three-panel drag-to-resize layout | ~8KB | **YES** | Custom CSS `resize` + drag handlers. Painful to build correctly, especially with keyboard a11y and min/max constraints. Library is small and focused. |
| `lucide-react` | Icon set | ~tree-shakes | **YES** | Inline SVGs. But Lucide tree-shakes well and provides 1000+ consistent icons. Manual SVGs for 20-30 icons is viable but tedious. |
| `@radix-ui/*` (via shadcn/ui) | Accessible primitives (Dialog, Accordion, Toggle, etc.) | Varies per component | **YES** | Build from scratch. But accessibility (keyboard nav, ARIA, focus trapping) is genuinely hard. Radix does it right. Note: shadcn/ui copies source files into your project — no runtime dependency on shadcn itself. |
| `clsx` | Conditional className strings | ~0.5KB | **YES** | Template literals. But `clsx` is tiny and prevents className bugs. |
| `tailwind-merge` | Resolves Tailwind class conflicts | ~3KB | **YES** | Without it, `bg-red-500 bg-blue-500` keeps both classes. Needed when merging user config styles with defaults. |
| `class-variance-authority` | Component variant definitions | ~2KB | **OPTIONAL** | Can use simple if/switch in component props. CVA is nice-to-have for shadcn/ui patterns but not required. |

### 3b. Data & IDs

| Package | Purpose | Size | Recommended? | Alternative |
|---------|---------|------|:---:|-------------|
| `uuid` | Unique IDs for sections and array items | ~3KB | **NO — use crypto.randomUUID()** | `crypto.randomUUID()` is built into all modern browsers and Node 19+. Zero dependency needed. |

### 3c. Drag & Drop (Level 2)

| Package | Purpose | Size | Recommended? | Alternative |
|---------|---------|------|:---:|-------------|
| `@dnd-kit/core` + `@dnd-kit/sortable` | Section reorder via drag | ~20KB combined | **OPTIONAL** | **Alt 1:** HTML5 Drag and Drop API — native, no library, but quirky on mobile and limited styling. Sufficient for desktop-only tool. **Alt 2:** Simple up/down arrow buttons — zero library, zero complexity. Users click arrows to reorder. Perfectly functional. **Recommendation:** Start with arrow buttons (L2 P2.2). Add @dnd-kit later if arrow UX feels limiting. |

### 3d. Spec Export (Level 3)

| Package | Purpose | Size | Recommended? | Alternative |
|---------|---------|------|:---:|-------------|
| `jszip` | Client-side zip creation for spec download | ~45KB | **NO — not needed** | **Alt 1:** Download individual files (config.json, spec.md, aisp.txt) as separate downloads. **Alt 2:** Single combined JSON file containing all specs. **Alt 3:** Copy-to-clipboard for the spec package (paste into Claude Code). **Alt 4:** Blob download of a single concatenated markdown file. **Recommendation:** Use individual file downloads or a "Copy All for Claude Code" button. A zip is over-engineering for a capstone demo. |

### 3e. Future (Level 4+, Beyond MVP)

| Package | Purpose | Level | Mandatory? | Alternative |
|---------|---------|-------|:---:|-------------|
| `@supabase/supabase-js` | Auth + database + real-time | L4 | YES for L4 | Firebase, but Supabase is already specified and offers RLS + JSONB. |
| Anthropic Claude SDK | LLM integration | L5 | YES for L5 | Direct fetch to API. SDK is optional but adds types + streaming. |
| Web Speech API | Voice STT | L6 | Built-in | Whisper API as fallback — but that requires server-side processing. |

## 4. EXPLICITLY FORBIDDEN

Per the Swarm Protocol (Document 5, Section 5.4):

| Package | Why Forbidden |
|---------|---------------|
| `redux` | Zustand is mandated (ADR-005), Redux adds boilerplate |
| `axios` | `fetch()` is built in |
| `lodash` | Use native JS methods |
| `moment` / `dayjs` | Use `Intl.DateTimeFormat` or native Date |
| `styled-components` / `emotion` | Tailwind is mandated |
| `formik` / `react-hook-form` | Simple controlled inputs + Zod sufficient |
| `monaco-editor` | Too heavy (~5MB) for JSON editing. Use a simple textarea with syntax highlighting |
| `next` / `remix` / `astro` | Vite SPA is mandated (ADR-002) |
| `@tanstack/react-query` | No server state management needed until L4+ (and even then, Supabase handles it) |

## 6. HOSTING OPTIONS

The app is a static SPA (client-side only through L3). Any static host works.

| Option | Cost | Rationale |
|--------|------|-----------|
| **Vercel** | Free tier | Zero-config Vite deployment, automatic previews |
| **Netlify** | Free tier | Same as Vercel, slightly different DX |
| **GitHub Pages** | Free | Already on GitHub, simplest possible deploy |
| **Supabase Hosting** | Included with project | If using Supabase for L4+, consolidate |
| **Cloudflare Pages** | Free tier | Fast CDN, generous limits |

**Recommendation:** Vercel or Netlify for instant deploys. GitHub Pages if minimizing external services.

For L4+ (Supabase backend), you'll need Supabase anyway. Host the SPA on Vercel/Netlify and connect to Supabase for auth/data.

---

## 7. DECISION SUMMARY

| Decision | Choice | Impact |
|----------|--------|--------|
| Drop `uuid` | Use `crypto.randomUUID()` | -1 dependency |
| Drop `jszip` | Individual downloads or clipboard | -1 dependency, -45KB |
| Drop `class-variance-authority` | Conditionals in components | -1 dependency |
| Defer `@dnd-kit/*` | Arrow buttons first | -2 dependencies until needed |
| Keep `react-resizable-panels` | Too hard to build well | Worth the 8KB |
| Keep `lucide-react` | Tree-shakes, consistent icons | Worth it vs manual SVGs |
| Keep Radix primitives | Accessibility is hard | Copy-paste via shadcn/ui |

**Net result: 13 production deps (down from 18+ in original spec), ~50KB gzipped runtime.**
