# ADR: End-of-Phase Close Protocol

**Status:** ACCEPTED  
**Date:** 2026-04-05  
**Author:** Bradley Ross  
**Context:** Every phase of Hey Bradley must follow a formal close protocol to maintain quality, traceability, and continuity across sessions.

---

## Decision

Every phase MUST produce the following artifacts before it can be marked CLOSED. No exceptions. The phase score cannot be assigned until all artifacts exist.

## End-of-Phase Artifacts (Required)

### 1. Session Log (`phase-N/session-log.md`)
- Every action taken, numbered sequentially
- Commit hashes for each change
- Decisions made with rationale
- Issues found and how they were resolved
- Metrics tracking (start vs. end of phase)

### 2. Living Checklist (`phase-N/living-checklist.md`)
- Every deliverable from the phase scope
- Marked [x] for complete, [ ] for incomplete
- Deferred items explicitly noted with destination phase
- Updated in real-time during the phase (not backfilled)

### 3. Retrospective (`phase-N/retrospective.md`)
- Phase score (composite /100)
- Scores by dimension (Visual, Functionality, Code Quality, Content Richness, DX — each /20)
- Historical score table (all phases)
- What shipped (with evidence)
- What didn't ship (honest, with reasons)
- Key decisions made
- Lessons learned
- Carry-over to next phase

### 4. Wiki Page (`wiki/NN-phase-N-title.guide.html`)
- Public-facing summary of the phase
- Matches existing wiki page format (dark theme, CSS variables)
- Links to previous and next phase wiki pages
- Badges showing status and score

### 5. CLAUDE.md Update
- Current Phase status (CLOSED with score)
- Updated metrics (tests, examples, effects, lines, etc.)
- Phase Roadmap table updated
- No other sections changed

### 6. Next-Phase Preflight (`phase-N+1/README.md`)
- Grounding report (current asset counts, test count, key files)
- Sprint breakdown with effort estimates
- ADRs needed (listed with titles and sprint assignments)
- Testing plan (new tests per sprint, total target)
- Dependencies and carry-over from current phase
- Enforcement section (references this ADR)

---

## Phase Score Methodology

| Dimension | Weight | Criteria |
|-----------|--------|----------|
| Visual Quality | /20 | Does it look like a funded startup? Polish, animations, consistency |
| Functionality | /20 | Do all features work? No broken buttons, toggles, dead controls |
| Code Quality | /20 | TypeScript clean, no as-any, schemas validated, tests passing |
| Content Richness | /20 | Examples, images, themes, effects, specs — breadth and depth |
| Developer Experience | /20 | Resources, docs, JSON visibility, AISP guide, cross-references |

**Score thresholds:**
- 90+: Exceptional — ship it
- 80-89: Strong — minor polish needed
- 70-79: Good — some gaps to address
- 60-69: Acceptable — significant items deferred
- Below 60: Needs work — should not close without fixing P0s

---

## Consequences

- **Positive:** Full traceability across sessions. Any future AI agent can read the session log and understand exactly what happened, why, and what's next.
- **Positive:** Honest scoring prevents "done-but-broken" phases.
- **Positive:** Next-phase preflight means every session starts grounded.
- **Negative:** ~30-60 min overhead per phase for documentation. Worth it for a capstone project.

---

## Alternatives Considered

1. **No formal close** — Rejected. Led to drift in early phases where decisions weren't captured.
2. **Minimal close (just a commit message)** — Rejected. Not enough context for future sessions.
3. **Heavy close (full audit report)** — Deferred to Phase 17/18 which define the formal feature review process.
