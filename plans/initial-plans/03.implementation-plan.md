## LEVEL 1: CORE BUILDER

### Phase 1.0 — Shell & Navigation (Day 1)

**Goal:** Vite site running. Every navigation element works. Every page/tab has a placeholder. The shell looks like a finished product.

**Definition of Done:**
- Three-panel layout renders with warm cream chrome
- LISTEN/BUILD and DRAFT/EXPERT toggles switch UI state
- All 4 center tabs (REALITY, DATA, XAI DOCS, WORKFLOW) navigable
- Status bar shows "READY AISP SPEC V1.2" and "MODE: DRAFT CONNECTED"
- Chat input bar renders below left panel
- All content is placeholder/static — no data wiring

**Deliverables:**

| # | Component | Details |
|---|-----------|---------|
| 1 | Vite + React + TS + Tailwind + shadcn scaffold | App runs at localhost |
| 2 | Path aliases (`@/*`) | tsconfig + vite.config |
| 3 | `AppShell.tsx` | Top bar: HB logo, project name, V1.0.0-RC1 badge, device toggles, Share/Publish |
| 4 | `ModeToggle.tsx` | LISTEN/BUILD pill (left) + DRAFT/EXPERT pill (right) |
| 5 | `PanelLayout.tsx` | Resizable three-panel with react-resizable-panels |
| 6 | `TabBar.tsx` | REALITY, DATA, XAI DOCS, WORKFLOW — with active orange underline |
| 7 | `StatusBar.tsx` | Bottom bar, monospace, READY/MODE indicators |
| 8 | `ChatInput.tsx` | Mic icon + text input + send button below left panel |
| 9 | Draft left panel placeholder | Vibe cards (Warm/Ocean/Forest), section list |
| 10 | Expert left panel placeholder | Property inspector with hardcoded values |
| 11 | Draft right panel placeholder | Headline input, Layout selector cards, "More options →" |
| 12 | Expert right panel placeholder | Layout/Content/Style accordions |
| 13 | All 4 tab placeholders | Static content per tab |

**Acceptance Criteria:**
- [ ] `npm run dev` renders the complete shell
- [ ] DRAFT ↔ EXPERT toggles change left and right panels
- [ ] LISTEN ↔ BUILD toggles (Listen just shows placeholder for now)
- [ ] All 4 tabs navigate with active state styling
- [ ] Status bar renders with monospace text
- [ ] Zero console errors
- [ ] **Looks like a real product** (not scaffolding)

### Phase 1.1 — Hero + JSON Core Loop (Days 2-3)

**Goal:** Hero section renders from JSON. Controls change JSON. JSON tab shows live state. The bidirectional loop works.

**Definition of Done:**
- Hero section renders in REALITY tab from configStore
- Draft controls (vibe picker, headline input) update configStore → hero re-renders
- Expert controls (Direction, Align, Padding, Gap) update configStore → hero re-renders  
- DATA tab shows full JSON with syntax highlighting, live updates
- Editing JSON in DATA tab updates hero

**Deliverables:**

| # | Component | Details |
|---|-----------|---------|
| 1 | Zod schemas: `masterConfigSchema`, `heroSectionSchema` | In `src/lib/schemas/` |
| 2 | `configStore.ts` (Zustand) | With applyPatch, setTheme, section CRUD |
| 3 | `HeroCentered.tsx` | Renders from configStore, matching mockup styling |
| 4 | `DataTab.tsx` | JSON viewer with orange syntax highlighting, line numbers, COPY/EXPORT |
| 5 | Draft left panel wired | Vibe picker → theme cascade, section click → right panel context |
| 6 | Draft right panel wired | Headline input, Layout visual selector → configStore |
| 7 | Expert left panel wired | Project Explorer tree, property inspector → configStore |
| 8 | Expert right panel wired | Layout, Content, Style accordions → configStore |
| 9 | Bidirectional JSON sync | DATA tab editable → validates → merges → preview updates |

**Acceptance Criteria:**
- [ ] Clicking "Warm" vibe changes hero colors
- [ ] Typing a headline in right panel updates hero text immediately
- [ ] Expert property values (gap, padding) update hero layout
- [ ] DATA tab reflects all changes in real-time
- [ ] Editing DATA tab JSON updates preview and controls
- [ ] Preview re-render < 100ms after any change

## LEVEL 2: FULL SITE BUILDER

### Phase 2.1 — Vibe Onboarding

**Goal:** Landing page with clickable vibe cards. Selecting a vibe loads the builder.

**Deliverables:** Onboarding page with 6+ vibe cards (live mini-previews), "Describe your site" textarea (placeholder for L5.4), "Start from scratch" option, vibe switcher within builder.

### Phase 2.2 — All Core Sections

**Goal:** All 8 section types with 3-5 variants each. Section CRUD (add/remove/reorder).

**Deliverables:** Section registry, per-section schemas/renderers/controls, section add (dropdown), remove, duplicate, drag-and-drop reorder (@dnd-kit), section navigator in left panel, full-page preview mode.

**Acceptance Criteria:**
- [ ] All 8 section types render from JSON config
- [ ] Sections addable/removable/reorderable
- [ ] Full-page preview shows complete stacked site
- [ ] Expert mode shows section-specific property inspectors

### Phase 2.3 — Builder UX Polish

**Goal:** "Grandson can show grandma everything." Simpler than Wix for basic operations.

**Deliverables:** Section sidebar navigator, section highlight on select, quick actions (move up/down, duplicate, delete), global undo/redo, named project save/load, first-time tooltips.

### Phase 3.1 — Pillar Docs (XAI Docs Tab)

**Goal:** Auto-generate North Star, Architecture, and Implementation Plan from current config. Rendered in XAI DOCS tab.

**Deliverables:** Spec templates as template-literal functions, HUMAN view (structured markdown), AISP view (AISP Crystal Atom format), copy/export buttons, live updates when config changes.

**Deliverables:** Per-section spec generators, download zip (pillar docs + section specs + config.json), Claude Code formatted output.

## LEVEL 4: AUTH & DATABASE

### Phase 4.1 — Supabase Auth + Database
Login/signup, project dashboard, swap LocalStorageAdapter for SupabaseAdapter, auto-save to Supabase.

### Phase 4.2 — Templates from Database
Vibe presets and section templates from database instead of hardcoded files.

### Phase 4.3 — LLM Pillar Doc Generation
Paste requirements → LLM generates pillar docs + selects vibe + populates sections.

---

## LEVEL 5: LLM FUNCTIONALITY

### Phase 5.1 — Chat Bot (Hero Only)
Chat input → LLM → JSON patch → hero updates. Simple changes under 2 seconds.

### Phase 5.2 — Copy Suggestions
Click section → type changes → LLM suggests 3 copy options with best-practice rules.

### Phase 5.3 — Section Inference
LLM infers target section from text alone. Two-step flow: classify intent → generate patch.

### Phase 5.4 — Onboarding Purpose
"Describe your website" textarea → LLM parses → selects vibe → populates all sections.

---

## LEVEL 6: VOICE MODE

### Phase 6.1 — Microphone Button
Web Speech API STT. Click to record, transcript populates chat input.

### Phase 6.2 — Listen Mode (Task Queue)
Continuous listening. Every ~15s: LLM Call 1 translates speech → task list. User approves. LLM Call 2 generates patches.

### Phase 6.3 — Virtual Whiteboard
Auto-apply mode. No approval gate. Site updates every ~15s as user talks.

---

## LEVEL 7: ENTERPRISE SPECS

### Phase 7.1 — AISP Mode
Toggle enables AISP Crystal Atom prompts for all LLM calls. Measurably lower error rate.

### Phase 7.2 — Change Logs
`project_versions` table with JSON diffs. Version timeline with rewind/restore.

### Phase 7.3 — Full Human Specs
Complete documentation page. Print-friendly. "Copy All for Claude Code" button.

### Phase 7.4 — AI-First AISP Export
AISP-formatted spec package. Validation scores per section. < 2% ambiguity.
