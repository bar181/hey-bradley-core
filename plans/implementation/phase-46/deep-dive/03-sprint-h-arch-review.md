# Sprint H End-of-Sprint — R3 Architecture Review (Lean)
> **Score:** 91/100
> **Verdict:** PASS

## Summary
Sprint H lands a coherent reference-ingestion subsystem: two symmetric kv-chunked repositories, two upload widgets, one summary panel, two Λ extensions (CONTENT_ATOM.brand_voice, INTENT_ATOM.project_context), and three ADRs that cross-reference each other cleanly. LOC discipline holds (every file under the 500 cap), DDD boundaries are clean, and Σ widths are intentionally unchanged so existing validators stay strict — but one ADR↔code enum drift (ADR-068) and one asymmetric injection path (brand context read through `buildSystemPrompt`, codebase context read through `chatPipeline`) are real architectural smells that should be reconciled before Sprint I.

## MUST FIX
- **F1 — ADR-068 enum drift vs `codebaseContext.ts` ProjectType.** ADR-068 §"Project-type heuristic" documents the closed enum as `'saas-app' | 'marketing-site' | 'docs-site' | 'portfolio' | 'unknown'`, but `codebaseContext.ts:35-40` ships `'saas-app' | 'landing-page' | 'static-site' | 'portfolio' | 'unknown'`. INTENT_ATOM (`intentAtom.ts:84-90`), `PROJECT_TYPE_TARGET_BIAS`, `chatPipeline.ts:211-219` defensive type guard, and `CodebaseContextUpload.projectTypeLabel` ALL reflect the code shipped — only the ADR is wrong. Either amend ADR-068 to match the code, or rename the four enums + update the bias table. Picking the ADR amend is faster and matches what's in production.

## SHOULD FIX
- **L1 — Asymmetric injection sites for the two reference channels.** Brand context is read directly inside `prompts/system.ts:8` via static `import { readBrandContext }` from the repo. Codebase context is read inside `chatPipeline.ts:204-224` via DEFENSIVE dynamic `import()` + try/catch. The reasons are real (system.ts wants the body, chatPipeline wants the cheap manifest), but a single `loadReferenceContext()` helper exposing `{brand, codebase}` with one try/catch each would normalize the surface and remove the "where do I read these?" friction for the next reference channel. **Defer if Sprint I adds a third reference; promote to F2 if not.**
- **L2 — Manifest+chunk pattern is duplicated, not extracted (P37 R3 L1 echo).** `brandContext.ts` (127 LOC) and `codebaseContext.ts` (chunking part — first ~150 LOC excl. detector) carry near-identical `chunkKey`, `read*Manifest`, `read*`, `write*`, `clear*`. Two copies is healthy KISS; a third channel (Sprint I uploads, conversation memory, etc.) makes this WET. A generic `kvChunkedRepo<TManifest>(prefix, validate)` helper would eliminate ~100 LOC at zero behavior-risk. Not blocking — flag now so the third copy doesn't ship.
- **L3 — `GROUNDING.md` is stale.** Header "Last updated: 2026-04-28 (mid-Sprint F P37, pre-seal)"; §2 still says "Sprint F P2 (Phase 37) — mid-flight, pre-seal". Sprint H/P44-P46 trajectory is not captured. The wiki (`llm-call-process-flow.md`) DOES claim "Last verified P46" and references ADR-067/068/069 — that doc is current; GROUNDING.md alone is the laggard. Update at seal.
- **L4 — `ReferenceManagement` re-implements the same clear-with-confirm flow already in the upload widgets.** Each widget owns a `handleClear` (window.confirm + clear* + refresh manifest). `ReferenceManagement.handleClearBrand/Codebase` does the same thing with its own state counter. Two divergent code paths to clear the same kv rows is fine for MVP, but ADR-069 already calls it out as deferrable to a "shared store at Sprint I" — explicitly link the two with a small `useReferenceManifests()` hook when the third channel lands.

## Acknowledgments
- **A1 — LOC discipline holds across the cumulative sprint.** Largest files: `CodebaseContextUpload.tsx` (406), `chatPipeline.ts` (378), `BrandContextUpload.tsx` (228), `codebaseContext.ts` (222). All comfortably under the 500-LOC hard cap. The split between widget (extract+detect+UI) and repo (typed CRUD) is exactly the right cut.
- **A2 — DDD bounded contexts are clean.** Both reference repos sit in `persistence/repositories/` next to the existing 8 repos (projects, sessions, messages, kv, examplePrompts, llmCalls, llmLogs, userTemplates). No drift — the directory continues to mean "typed CRUD over a kv or table". UI consumers in `components/settings/`, intelligence consumers in `intelligence/aisp/` + `intelligence/prompts/`. No cross-context import smells.
- **A3 — Σ widths preserved on both Λ extensions.** CONTENT_ATOM Λ.brand_voice (P44 / ADR-067) and INTENT_ATOM Λ.project_context (P45 / ADR-068) are both purely additive logistics-layer signals. Σ-shape validators (`validateGeneratedContent` V1-V4, `classifyIntent` Γ R1-R4) are unchanged; V5 / bias-rank only re-assert existing constraints when the optional channel is present. Pattern is coherent; tests prove byte-identical fall-through when channel absent.
- **A4 — ADR cross-reference web is consistent.** ADR-067 mentions "P45 reuses the same persistence + injection pattern documented here". ADR-068 cites ADR-067 as "sibling pattern; same kv-chunked motif". ADR-069 cites both as the lifecycle predecessors. All three list ADR-040/046 as the shared infrastructure ground. Sprint H reads as a designed three-phase arc, not three drive-bys.
- **A5 — Settings drawer mount order is correct.** `ReferenceManagement` (summary, manifest-only reads) → `BrandContextUpload` → `CodebaseContextUpload`. Summary first answers "what do I have?"; uploads below answer "add or replace". User-flow rationale is documented in ADR-069 §Decision. Backwards mount would be wrong.
- **A6 — `validateGeneratedContent` cost is negligible.** The unconditional Ε-validation pass on every generation is six ~O(1) string/enum checks plus one `isCleanContent` regex scan. ~µs-scale. The strict P31 output contract is well worth the discipline; this is correctly paranoid, not overcautious.
- **A7 — AISP barrel (`aisp/index.ts`) is non-circular.** P45 additions (`PROJECT_TYPES`, `PROJECT_TYPE_TARGET_BIAS`, `ProjectType`) all source from `intentAtom` (a leaf module). The barrel re-exports outward; no module imports the barrel from inside the aisp/ tree. Safe.
- **A8 — Test architecture spread is sensible.** 68 spec files total / 3 new for Sprint H (`p44-brand-upload`, `p45-codebase-ref`, `p46-reference-management`). One spec per phase mirrors the Sprint D/E pattern. Cumulative 408+ pure-unit cases per the master ledger; Sprint H's incremental count is in line with prior phases.
- **A9 — `readBrandContext`/`readCodebaseContext` "refuse partial joins" semantics.** Both repos return `null` when ANY chunk is missing rather than silently joining a truncated string. Correct by-default safety — corrupt kv state surfaces as an absent reference, not a wrong one.
