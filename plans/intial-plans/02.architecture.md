## 1. CORE PRINCIPLE: JSON AS SINGLE SOURCE OF TRUTH

Everything in Hey Bradley derives from a single JSON state object. The UI is a renderer. The builder is a JSON editor. The LLM is a JSON transformer. The specs are JSON interpreters.

```
                    ┌──────────────────────┐
                    │    MASTER JSON       │
                    │  (configStore)       │
                    └──────────┬───────────┘
                               │
         ┌─────────────────────┼─────────────────────┐
         ▼                     ▼                     ▼
  ┌─────────────┐      ┌─────────────┐      ┌─────────────┐
  │  REALITY    │      │  DATA       │      │  XAI DOCS   │
  │  Tab        │      │  Tab        │      │  Tab        │
  │  (Preview)  │      │  (JSON)     │      │  (Specs)    │
  └─────────────┘      └─────────────┘      └─────────────┘
         ▲                     ▲                     ▲
         │                     │                     │
  ┌─────────────┐      ┌─────────────┐      ┌─────────────┐
```

## 2. TWO-MODE ARCHITECTURE

### 2.1 Toggle System

Hey Bradley has **two independent toggle axes** producing four composite UI states:

```
AXIS 1: INTERACTION MODE
  LISTEN ←→ BUILD
  (How input enters the system)

AXIS 2: COMPLEXITY MODE  
  DRAFT ←→ EXPERT
  (How much control is exposed)
```

```aisp
⟦
  Ω := { Define UI mode state machine }
  Σ := { InteractionMode:{"LISTEN"|"BUILD"}, ComplexityMode:{"DRAFT"|"EXPERT"}, UIState:InteractionMode×ComplexityMode }
  Γ := { R1: ∀ state∈UIState : config_data_unchanged(state), R2: IF mode_switch THEN only_controls_change, R3: ∀ state : same_configStore_reads }
  Λ := { default_interaction:="BUILD", default_complexity:="DRAFT" }
  Ε := { V1: VERIFY switching modes preserves all JSON state, V2: VERIFY DRAFT↔EXPERT shows identical preview }
⟧
```

### 2.2 Four Composite States

| State | Left Panel | Center Canvas | Right Panel |
|-------|-----------|---------------|-------------|
| **BUILD + DRAFT** | Vibe picker, section list, "Start Talking" button, chat input | Reality tab (preview with section selection borders) | Simplified: Headline, Layout visual selector, "More options →" |
| **BUILD + EXPERT** | Project Explorer tree (HeroSection > Title, Subtitle, PrimaryCTA), accordion property inspector | Reality tab (preview with section selection borders) | Full inspector: Layout (Direction, Align, Padding, Gap), Content (Headline, Subtitle, Show toggle), Style (Background, Text Color with CSS variables) |
| **LISTEN + DRAFT** | Dark overlay, pulsing red orb, "START LISTENING" button | Dark overlay, larger ambient red orb | Hidden |
| **LISTEN + EXPERT** | Dark overlay, pulsing red orb, "START LISTENING" button | Dark overlay, larger ambient red orb | Hidden |

### 2.3 Center Canvas: Four Tabs

The center panel is not just a preview — it is a multi-view workspace with four tabs:

| Tab | Icon | Content | Updates From |
|-----|------|---------|-------------|
| **REALITY** | ⊞ | Live preview of website sections, click-to-select with dashed orange borders | configStore changes |
| **DATA** | ⊟ | Full JSON with syntax highlighting, line numbers, COPY + EXPORT, editable | configStore (bidirectional) |
| **XAI DOCS** | ⊕ | HUMAN/AISP sub-toggle, generated specification documents | configStore (derived) |
| **WORKFLOW** | ⊗ | LLM pipeline: Voice Capture → Intent Parsing → AISP Gen → Schema Validation → Reality Render → Edge Deploy, live stream output log | LLM processing events |

## 3. THREE-LAYER ARCHITECTURE

```
LAYER 1: INPUT (How changes enter)
├── DRAFT Controls (vibe picker → macro mutations affecting 15+ JSON nodes)
├── EXPERT Controls (property inspector → micro mutations targeting single JSON nodes)
├── JSON Editor (Data tab → direct JSON editing with Zod validation)
├── Chat Input (text → LLM → JSON patch)
├── Voice / Listen Mode (speech → STT → LLM → JSON patches)
└── Import (JSON file upload → validation → configStore load)

LAYER 2: STATE (Single source of truth)
├── configStore (Zustand) — Master JSON config
│   ├── spec: "aisp-1.2"
│   ├── page: "index"
│   ├── version: "1.0.0-RC1"
│   └── sections: [{ type, id, layout, content, style }]
├── uiStore (Zustand) — UI mode state (LISTEN/BUILD, DRAFT/EXPERT, activeTab, selectedSection)
├── chatStore (Zustand) — Chat/voice message history and pending patches
└── Storage Abstraction → LocalStorage (L1-L3) | Supabase (L4+)

LAYER 3: OUTPUT (What the system produces)
├── REALITY Tab — React components rendered from configStore
├── DATA Tab — Formatted JSON from configStore
├── XAI DOCS Tab — Generated markdown/AISP specs from configStore
├── WORKFLOW Tab — Pipeline status from LLM processing events
├── JSON Export — Downloadable config file
└── Spec Package — Zip of pillar docs + section specs + JSON
```

## 6. ARCHITECTURE DECISION RECORDS (ADR)

### ADR-001: JSON as Single Source of Truth
**Decision:** All site configuration stored as a single JSON object in a Zustand store. Every input method reads from and writes to this same object.  
**Consequence:** No hidden state in components. JSON tab, spec generator, and preview are all derived views.

### ADR-002: Vite + React (not Next.js)
**Decision:** Use Vite + React for a client-side SPA. No SSR needed — Hey Bradley produces specs, not hosted sites.  
**Consequence:** All client-side. LLM calls via proxy or Supabase Edge Functions.

### ADR-003: CSS Values in JSON (not Tailwind Classes)
**Decision:** Store CSS values (`"24px"`, `"flex"`, `"#faf8f5"`) in JSON, not Tailwind utility classes. The Expert mode property inspector shows raw CSS values (matching mockup Image 7).  
**Consequence:** Preview components translate JSON CSS values to Tailwind classes or inline styles at render time. Spec export shows exact CSS values. LLM patches use CSS values.

### ADR-004: Section-Based Architecture
**Decision:** Primary unit is a "section" (Hero, Features, Pricing), not a generic component.  
**Consequence:** Each section type has its own schema, variants, and default values.

### ADR-005: Zustand (not Redux/Context)
**Decision:** Zustand for all state management. Three stores: `configStore` (site data), `uiStore` (mode state), `chatStore` (messages).  
**Consequence:** No providers, no boilerplate. Middleware handles persistence and undo.

### ADR-006: Deferred Database (LocalStorage → Supabase)
**Decision:** Phases 1-3 use localStorage via `IProjectRepository`. Phase 4 swaps adapter.  
**Consequence:** Storage abstraction designed from Phase 1.

### ADR-007: LLM Returns JSON Patches
**Decision:** LLM returns partial JSON patches, not full state. Client deep-merges.  
**Consequence:** Arrays replace entirely (never concat). Objects merge recursively. Null deletes keys.

### ADR-008: Progressive Disclosure via DRAFT/EXPERT
**Decision:** Two UI modes rather than layered progressive disclosure within a single panel. DRAFT hides all granular CSS. EXPERT reveals everything.  
**Consequence:** Same data, different controls. Mode toggle is the single switch between simplicity and power.

### ADR-009: Warm Light Chrome (not Dark)
**Decision:** Builder chrome uses warm cream palette (`#faf8f5` background, `#e8772e` accent, `#2d1f12` text). Only Listen mode goes dark.  
**Consequence:** Warm, approachable, editorial aesthetic. Monospace uppercase labels for engineering credibility. Dark mode reserved for theatrical Listen mode contrast.

### ADR-010: Two-Step LLM Flow
**Decision:** Step 1 classifies intent/target section (fast, <500ms). Step 2 generates JSON patch for that section (focused, <1.5s).  
**Consequence:** Two API calls, but smaller prompts → less hallucination, auditable intent classification.

### ADR-011: Four Center Tabs (not Separate Routes)
**Decision:** Reality, Data, XAI Docs, and Workflow are tabs within the builder, not separate pages/routes.  
**Consequence:** Single-page builder. Tab content updates reactively from configStore. No route navigation to see specs — they're always one tab away.

## 8. CROSS-CUTTING CONCERNS

### 8.1 Error Handling

| Boundary | Strategy | UX |
|----------|----------|-----|
| Component render crash | React Error Boundary per section | "This section had an issue" + repair button |
| Invalid JSON edit | Zod validation before merge | Red highlight + error in DATA tab |
| LLM invalid patch | Schema validation → reject + retry | "Couldn't understand, try again" |
| Network failure (L4+) | Optimistic UI + retry queue | Work continues offline |

### 8.2 Performance Targets

| Metric | Target |
|--------|--------|
| Config change → visual update | < 100ms |
| Vibe switch (all sections) | < 200ms |
| LLM response (simple) | < 2 seconds |
| Initial page load | < 1.5 seconds |
| Listen mode cycle | ~15 seconds |

### 8.3 Security (Phase 4+)

| Concern | Approach |
|---------|----------|
| XSS in preview | React auto-escaping, no dangerouslySetInnerHTML |
| Auth | Supabase RLS — users only access own projects |
| JSON injection | Zod validation on all config writes |
| LLM prompt injection | User input treated as data, not instructions |
| API keys | Server-side proxy, no client-side keys |
