## 1. THE UX-FIRST EXECUTION SEQUENCE

### 1.0 Implementation Philosophy

Bradley's build sequence is:

```
1. Get the Vite site running (scaffold, routing, shell)
2. Get the JSON and hero sections working (core loop)
3. Placeholders for all pages/tabs (full navigation works)
4. Best-in-class design for every visible component (wow factor)
```

This means: **ship a clickable, navigable, visually polished shell before wiring any backend logic.** Static mockups with hardcoded data are perfectly acceptable at every early step. The moment something is visible to a human reviewer, it must look production-grade.

### 1.1 The Two-Mode Architecture

Hey Bradley has a **two-mode** UI architecture that governs every screen. This is NOT a three-panel accordion builder. It is a modal interface with two personality states:

```
┌─────────────────────────────────────────────────────────────────┐
│  [LISTEN │ ● BUILD]          [DRAFT │ EXPERT]                   │
├──────────┬──────────────────────────────────────┬───────────────┤
│          │  [REALITY] [DATA] [XAI DOCS] [WORKFLOW]              │
│  LEFT    │                                      │    RIGHT      │
│  PANEL   │         CENTER CANVAS                │    PANEL      │
│          │         (tabbed views)                │    (context   │
│          │                                      │     panel)    │
│          │                                      │               │
├──────────┴──────────────────────────────────────┴───────────────┤
│  [🎤 Describe what to build...]                        [➤]     │
├─────────────────────────────────────────────────────────────────┤
│  READY  AISP SPEC V1.2                    MODE: DRAFT CONNECTED │
└─────────────────────────────────────────────────────────────────┘
```

**Two toggle axes — both are always visible:**

| Toggle | Left State | Right State | What Changes |
|--------|-----------|-------------|--------------|
| **LISTEN / BUILD** | Listen mode (AI voice capture) | Build mode (manual editing) | Left panel content + center canvas overlay |
| **DRAFT / EXPERT** | Simplified controls (grandma) | Granular controls (Framer user) | Left panel detail level + right panel visibility |

**These produce four composite states:**

| State | Left Panel | Center | Right Panel |
|-------|-----------|--------|-------------|
| BUILD + DRAFT | Vibe picker, section list, "Start Talking" | Reality tab (live preview) | Simplified: Headline, Layout, "More options →" |
| BUILD + EXPERT | Project Explorer tree, component tree | Reality tab (live preview) | Full property inspector: Layout, Content, Style |
| LISTEN + DRAFT | Pulsing red orb, "Start Listening" button | Dark overlay with larger red orb | Hidden or minimal |
| LISTEN + EXPERT | Pulsing red orb, "Start Listening" button | Dark overlay with larger red orb | Hidden or minimal |

## 2. THE TWO-MODE UI STRICT BOUNDARIES

### 2.1 Draft Mode Controls (Grandma View)

Draft mode hides all granular CSS properties. Controls operate on **macros** — high-level presets that cascade to multiple JSON nodes under the hood.

```
DRAFT LEFT PANEL:
├── "What do you want to build?" prompt
├── "Start Talking" button (voice onramp)
├── "Pick a vibe" — 3 preset cards (Warm, Cool/Ocean, Fresh/Forest)
│   └── Clicking a vibe mutates: theme.colors (all), theme.typography, 
│       section backgrounds, accent colors — 15+ JSON nodes from 1 click
├── "Your Sections" — ordered list with arrows
│   ├── Hero Section →
│   ├── Features →
│   ├── Pricing →
│   └── + Add Section
└── [no padding/gap/flex controls — those are Expert only]

DRAFT RIGHT PANEL:
├── Section name as header (e.g., "HERO", "Features", "Call to Action")
├── Headline — text input
├── Layout — visual selector (Left, Center, Right icons — matching Image 4)
├── "More options →" — link to Expert mode for this section
└── [no CSS values, no hex colors, no pixel inputs]
```

### 2.2 Expert Mode Controls (Framer View)

Expert mode reveals the full property inspector. Controls operate on **micro nodes** — individual JSON properties.

```
EXPERT LEFT PANEL:
├── "EXPERT EDITOR" header with gear icon
├── LAYOUT (accordion, expanded by default)
│   ├── DISPLAY         flex
│   ├── DIRECTION       column
│   ├── ALIGN           center
│   ├── JUSTIFY         start
│   ├── GAP             24px
│   ├── PADDING         64px
│   ├── MAX-WIDTH       1280px
│   ├── FULL-WIDTH      [toggle]
│   └── RESPONSIVE      [toggle]
├── CONTENT (accordion)
│   └── (text fields for all copy)
├── SECTIONS (accordion) — Hero, Features, CTA + "Add section"
├── IMAGES (accordion) — HERO-BG, LOGO with file references
├── STYLE (accordion)
│   ├── BACKGROUND      #faf8f5
│   ├── COLOR           #2d1f12
│   ├── FONT-FAMILY     Inter
│   ├── FONT-SIZE       16px
│   ├── LINE-HEIGHT     1.6
│   ├── BORDER-RADIUS   8px
│   ├── DARK-MODE       [toggle]
│   ├── ANIMATIONS      [toggle]
│   └── PALETTE         [color swatches]
└── SETTINGS (accordion, collapsed)

EXPERT RIGHT PANEL:
├── Section name header with settings icon
├── Layout section: Direction arrows, Align buttons, Padding input, Gap input
├── Content section: Headline textarea, Subtitle textarea, Show Subtitle toggle
├── Style section: Background color, Text Color (using var(--bg), var(--text))
└── [Full CSS property access]
```

### 2.3 The Shared Truth Rule

**Both modes read and write to the exact same Zustand `configStore`.** Mode switching only changes which controls are rendered — never the underlying data. Switching from DRAFT to EXPERT mid-session must show the same site state with more granular controls.

```typescript
// The mode state is UI-only — it lives in a separate uiStore, NOT in configStore
const useUIStore = create<UIStore>((set) => ({
  interactionMode: 'BUILD' as 'LISTEN' | 'BUILD',
  complexityMode: 'DRAFT' as 'DRAFT' | 'EXPERT',
  activeTab: 'REALITY' as 'REALITY' | 'DATA' | 'XAI_DOCS' | 'WORKFLOW',
  selectedSectionId: null as string | null,
  
  setInteractionMode: (mode) => set({ interactionMode: mode }),
  setComplexityMode: (mode) => set({ complexityMode: mode }),
  setActiveTab: (tab) => set({ activeTab: tab }),
  selectSection: (id) => set({ selectedSectionId: id }),
}));
```

## 3. THE ZOD-FIRST DIRECTIVE (Data Layer Laws)

### 3.1 The Rule

**No stateful component may read from or write to the configStore until its Zod schema exists.** Static visual prototypes with hardcoded data are allowed (and encouraged as a first pass). The moment a component connects to the store, its schema must be defined.

```typescript
// ✅ CORRECT — Single source of type truth
export const HeroContentSchema = z.object({
  heading: z.object({
    text: z.string().default('Build with voice.'),
    level: z.number().default(1),
    size: z.string().default('48px'),
    weight: z.number().default(700),
  }),
  subheading: z.string().default('Ship in minutes.'),
  cta: z.object({
    text: z.string().default('Get Started'),
    url: z.string().default('#pricing'),
  }),
});

export type HeroContent = z.infer<typeof HeroContentSchema>;

// ❌ FORBIDDEN — Manual duplicate interface
interface HeroContent { heading: { text: string; level: number } }
```

### 3.2 JSON Shape Must Match the Mockups

The JSON structure visible in the Data tab (Image 8) is the canonical shape. The Zod schema must produce JSON that looks like this:

```json
{
  "spec": "aisp-1.2",
  "page": "index",
  "version": "1.0.0-RC1",
  "sections": [
    {
      "type": "hero",
      "id": "hero-01",
      "layout": {
        "display": "flex",
        "direction": "column",
        "align": "center",
        "gap": "24px",
        "padding": "64px"
      },
      "content": {
        "heading": { "text": "Build with voice.", "level": 1, "size": "48px", "weight": 700 },
        "subheading": "Ship in minutes.",
        "cta": { "text": "Get Started", "url": "#pricing" }
      },
      "style": {
        "background": "#faf8f5",
        "color": "#2d1f12",
        "fontFamily": "Inter",
        "borderRadius": "8px"
      }
    }
  ]
}
```

### 3.3 Array ID Requirement

All array items must have unique `id` fields for LLM patch targeting:

```typescript
// ✅ CORRECT
const FeatureItemSchema = z.object({
  id: z.string(),
  icon: z.string(),
  title: z.string(),
});

// ❌ FORBIDDEN — No ID means LLM can't target items
z.array(z.string())
```

## 4. STATE MUTATION LAWS (The Zustand Contract)

### 4.1 What Lives Where

| State Type | Store | Examples |
|-----------|-------|----------|
| Site config (exported JSON) | `configStore` | Theme, sections, copy, styles |
| UI mode state | `uiStore` | DRAFT/EXPERT, LISTEN/BUILD, active tab, selected section |
| Chat history | `chatStore` | Messages, pending patches |
| Ephemeral UI | `useState` local | Dropdown open, accordion expanded, input focus, typewriter index |
| Auth (Phase 4+) | `authStore` | User session |

### 4.2 The Hard Rule

> **If changing the value would change the exported JSON, the generated specs, or the LLM context, it MUST live in `configStore`. No exceptions.**

### 4.3 The Merge Protocol

When applying an LLM-generated JSON patch:

```
MERGE RULES:
1. Objects → deep merge recursively
2. Arrays → REPLACE ENTIRELY (never concat, splice, or index-merge)
3. null values → DELETE the key
4. undefined → SKIP (keep existing)
5. Primitives → OVERWRITE
```

**Never deep-merge arrays.** The LLM returns the complete array with modifications. Partial array patches cause index collisions and data corruption.

### 4.4 Undo/Redo

Every mutation pushes previous state to history stack. Handled by `undoMiddleware` — agents must not implement custom undo logic. Stack depth: 100 states (FIFO eviction).

## 6. THE DEPENDENCY & IMPORT FIREWALL (DDD Enforcement)

### 6.1 Path Aliases

```json
{
  "@/*":          ["./src/*"],
  "@/store/*":    ["./src/store/*"],
  "@/contexts/*": ["./src/contexts/*"],
  "@/templates/*":["./src/templates/*"],
  "@/lib/*":      ["./src/lib/*"],
  "@/types/*":    ["./src/types/*"]
}
```

### 6.2 Import Firewall

```
ALLOWED:
  @/contexts/builder/      → @/store/*, @/lib/*, @/types/*, @/templates/*, @/presets/*
  @/contexts/intelligence/  → @/store/configStore, @/lib/*, @/types/*
  @/contexts/specification/ → @/store/configStore, @/lib/*, @/types/*, @/templates/*/schema.ts
  @/contexts/persistence/   → @/store/*, @/lib/*, @/types/*
  @/templates/*/            → @/lib/cn.ts ONLY (templates are pure renderers)

FORBIDDEN (instant reject):
  @/contexts/builder/       ❌ NEVER imports from @/contexts/intelligence/
  @/contexts/intelligence/  ❌ NEVER imports from @/contexts/builder/components/
  @/templates/*/            ❌ NEVER imports from any @/contexts/*
```

### 6.3 Builder ↔ Intelligence Communication

Via Zustand store subscriptions only. No direct function imports across the firewall.

### 6.4 Storage Abstraction

All load/save goes through `IProjectRepository`. Direct `localStorage.getItem()` in components is **forbidden**.

## 7. THE LISTEN MODE PROTOCOL

### 7.1 Visual Specification

When LISTEN is toggled active:

1. Both left panel and center canvas transition to near-black (`#0a0a0f`) with 300ms fade
2. A **pulsing red orb** appears — centered in the left panel, with a larger ambient glow in the center canvas
3. The orb is layered: solid red core (`bg-red-500`, 40px) → blurred mid-ring (`bg-red-500/30`, blur-xl, 120px) → ambient glow (`bg-red-500/10`, blur-3xl, 300px)
4. The orb pulses: `animate-pulse` on the core, slower breathing animation on the outer glow
5. "▶ START LISTENING" button renders below the left panel orb
6. The right panel hides or minimizes
7. Top bar and status bar remain visible (warm cream, not dark)

### 7.2 AI Persona Voice

When the AI processes speech or generates output, text renders via a **Typewriter** component:

```
PERSONA RULES:
- No conversational filler ("Sure!", "Great question!", "Let me think...")
- System-like brevity: "Parsing intent...", "Generating AISP...", "Applying Warm Vibe."
- Monospace font, warm orange color, appearing character-by-character
- Each statement is 2-8 words maximum
```

### 7.3 Workflow Pipeline Visibility

When the AI is processing (Listen mode or Chat input), the Workflow tab lights up sequentially:

```
PIPELINE STEPS:
1. Voice Capture      → "Captured 3 intents"
2. Intent Parsing     → "hero, features, cta"
3. AISP Generation    → "3 sections generated"
4. Schema Validation  → "Validating aisp-1.2..."  (ACTIVE = orange spinner)
5. Reality Render     → "Waiting for validation"   (WAITING = gray clock)
6. Edge Deploy        → "Ready on validation"      (WAITING = gray clock)

LIVE STREAM OUTPUT (below pipeline):
[00:01] Voice → "hero section, dark background"
[00:02] Intent parsed: type=hero, bg=#0a0a0f
[00:03] AISP generated: @section hero hero-01
[00:04] Validating schema... ▌  (cursor blinks)
```

## 8. QUALITY GATES & CARDINAL SINS

### 8.1 Cardinal Sins (Instant Reject)

1. Manual TypeScript interface duplicating a Zod schema
2. Config data in `useState` instead of `configStore`
3. Direct `localStorage` access outside `IProjectRepository`
4. Import crossing a DDD firewall boundary
5. Unapproved package installation
6. Custom `.css` file without human approval
7. Array deep-merge instead of full replacement
8. Builder chrome using dark colors (must be warm cream)
9. `// @ts-ignore` or `any` type
10. `useEffect` for state synchronization
11. Templates importing from `@/contexts/*`
12. Listen mode without the red orb effect (generic chat box)

### 8.2 When to Ask for Human Review

| Trigger | Why |
|---------|-----|
| Phase stop condition reached | Visual validation |
| Package not in approved list | Dependency approval |
| Design decision not in Docs 1-6 | Architectural call |
| LLM prompt design | Accuracy validation |
| Any component that "doesn't look right" | Wow factor gate |
| Performance > 200ms render | Priority decision |
