Why This Matters

Phase 1 proves that a non-technical user can produce a professional website specification without writing code, hiring a designer, or learning any tool-specific vocabulary. Every interaction — picking a theme, toggling a component, changing a font — writes validated JSON that an AI or developer can turn into a production site. The builder hit 77/100 quality in six sub-phases, and every architectural decision is documented in ADRs for full transparency.

🛠 Technical Architecture

Everything in the builder is driven by a single JSON document called MasterConfig. Controls change JSON, JSON drives the preview, and the Data Tab shows the live state. Here are the five architectural pillars:

  • JSON-Driven Everything flows through MasterConfig (Zod-validated). Controls change JSON → JSON drives preview → Data Tab shows state. A single source of truth for the entire builder.
  • 3-Level Hierarchy site (metadata) → theme (visual defaults) → sections (content + components). Defined in ADR-012. Clean separation means themes can replace everything without breaking content.
  • Component Config Each UI element has id, type, enabled, order, props. Toggle visibility per component. Defined in ADR-016. Grandma toggles a switch; developer sees a structured JSON node.
  • CSS Custom Props Theme palette → 7 CSS variables on the preview container. Tailwind classes reference vars. Theme switch = update 7 vars, and the entire preview re-paints instantly. ADR-021.
  • 6-Slot Palette bgPrimary, bgSecondary, textPrimary, textSecondary, accentPrimary, accentSecondary. Minimalist themes fill multiple slots with the same color for visual restraint. ADR-019.

✓ Benefits

Phase 1 was designed around one principle: if grandma cannot use it, it is not done.

  • ♥ Grandma-Friendly. No design vocabulary needed. Pick a theme, see a preview. Toggle components on/off. Edit text directly. Every interaction is visual, and the result is immediate.
  • 📋 Proof-Based Specifications. The AISP protocol ensures near-zero ambiguity in generated specs. What you see in the builder is exactly what a developer or AI receives as instructions.
  • {} Full JSON Control. Power users can edit the raw JSON in the Data Tab. Every change validates against Zod schemas. Invalid JSON shows an error toast — it never silently corrupts state.
  • 🎨 Theme as Full Replacement. Switching themes replaces layout, components, colors, fonts, images — not just CSS variables. A theme is a complete design system, not a color swap.
  • 📚 Research-Backed Design. 12 ADRs documenting every decision. Theme categories based on Webflow/Squarespace marketplace data. Nothing was guessed — everything was researched and recorded.

✍ The Grandma Test

Picture this: a woman in her seventies sits down at a laptop. She has never heard the word “wireframe.” She does not know what a color palette is. But she knows what she likes when she sees it. She clicks a card that says “Wellness” and the screen transforms — soft greens, rounded corners, a calming hero image of a sunrise over a lake. She smiles. She did not write a single line of code. She did not ask for help. She just pointed and chose.

That moment is the entire reason Phase 1 exists. Every architectural decision — the three-level JSON hierarchy, the six-slot palette system, the component-level toggles — was made in service of that single interaction. The builder does not ask users to understand design systems. It presents ten curated themes, each a complete design opinion. When the user switches from “Wellness” to “Startup,” the layout changes, the fonts change, the images change, the component visibility changes. It is not a skin swap. It is a complete replacement of every visual decision, happening in under 100 milliseconds.

Under the surface, a Zod-validated JSON document records every choice as structured data. That document is the specification — the bridge between what grandma sees and what a developer builds. The AISP protocol ensures the spec is unambiguous enough to hand directly to an AI code generator with near-zero interpretation needed. Phase 1 built the foundation. Phases 2 through 5 will extend it from a hero section to a complete landing page, then add voice interaction and cloud persistence. But the principle stays the same: if grandma cannot use it, it is not done.

📎 Additional Details

Phase 1 spans sub-phases 1.0 through 1.3d. The src/ directory contains all builder source code. Theme definitions live in JSON template files validated by Zod schemas. The 12 ADRs (Architecture Decision Records) are stored in plans/ and document every non-obvious choice — from why we chose a 6-slot palette (ADR-019) to why theme switching does full JSON replacement instead of CSS variable swaps (ADR-021).

Known limitation. Phase 1 only edits the hero section. Other sections (Features, CTA, Footer) exist in the JSON schema but do not have SIMPLE tab editors yet. That is the primary goal of Phase 2. Additionally, the builder is client-side only — no cloud persistence until Phase 5 adds Supabase integration.
